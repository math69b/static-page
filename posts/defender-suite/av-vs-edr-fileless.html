AV vs. EDR: Por que um Ataque Fileless Passou pelo Defender e Foi Bloqueado pelo MDE

T1059.001 T1620 T1027 T1055

Boa parte das organizações ainda opera com a mentalidade de que ter o Windows Defender ativado é suficiente para proteger endpoints. Este post vai mostrar, com evidências de laboratório, por que essa premissa está errada — e mais importante, o que um EDR como o Microsoft Defender for Endpoint (MDE) faz de diferente.

O teste foi direto: executar a mesma cadeia de ataque em dois hosts Windows 11. Um com apenas o Defender padrão. Outro com o MDE onboard, integrado ao Defender XDR. O resultado foi completamente oposto nos dois casos.

// contexto

O que é um Ataque Fileless?

Um ataque fileless é aquele que não grava arquivos executáveis em disco. Em vez de dropar um .exe e executá-lo — padrão que qualquer antivírus detecta por assinatura — o payload vive inteiramente na memória do processo.

O vetor mais comum é o PowerShell: já está presente em qualquer Windows, é confiável pelo sistema operacional, e permite carregar código arbitrário na memória. Para o antivírus tradicional, isso é praticamente invisível. Não há arquivo para escanear. Não há hash para comparar com uma base de assinaturas. A ameaça simplesmente não existe no modelo de detecção para o qual ele foi projetado.

Antivírus foi desenhado para um mundo onde malware vivia em disco. O Red Team moderno vive na memória.
// arquitetura

A Diferença Fundamental: Assinatura vs. Comportamento

Antes de ver os resultados do lab, é importante entender o que cada solução realmente faz — e onde cada uma para de funcionar.

Capacidade Windows Defender (AV) Defender for Endpoint (EDR)
Detecção por assinatura ✓ Sim ✓ Sim
Análise comportamental em tempo real ✗ Limitado ✓ Sim (sensor de kernel)
Monitoramento de memória (RW→RX) ✗ Não ✓ Sim (TiSense)
AMSI com análise semântica Básico ✓ Avançado (reconhece intenção)
Visibilidade de chamadas de API (ETW) ✗ Não ✓ Sim
Correlação de eventos (XDR) ✗ Não ✓ Sim (Defender XDR)
Resposta automatizada (AIR) ✗ Não ✓ Sim
Hunting com KQL ✗ Não ✓ Sim (Advanced Hunting)

O ponto central está no monitoramento comportamental de memória e de chamadas de API. O AV tradicional atua principalmente no sistema de arquivos. O EDR tem um sensor instalado no kernel que observa o comportamento do processo em tempo de execução — incluindo como ele usa memória, quais APIs chama, e em qual sequência.

// lab

O Laboratório: Mesma Ameaça, Dois Resultados

Ambiente de Teste

  • Host A: Windows 11, Windows Defender ativado, sem MDE onboard
  • Host B: Windows 11, MDE onboard, integrado ao Defender XDR
  • Ataque: Shellcode carregado via PowerShell diretamente na memória, sem tocar o disco
  • Framework de C2: Sliver (listener mTLS)

Visão Geral da Cadeia de Ataque

Sem entrar em detalhes que tornariam este post um manual de ataque, a cadeia seguiu os seguintes passos lógicos:

  1. Download do payload criptografado diretamente para a memória via PowerShell — sem gravação em disco
  2. Decodificação do payload em memória
  3. Alocação de uma região de memória com permissão de leitura/escrita (RW)
  4. Cópia do shellcode decodificado para essa região
  5. Mudança de permissão da região para execução (RW → RX)
  6. Execução do shellcode via callback de API legítima do Windows

A etapa 5 é o ponto crítico para qualquer EDR. A transição de uma página de memória de RW para RX contendo shellcode desconhecido é um dos principais Indicadores de Comprometimento (IoC) comportamentais monitorados por sensores de kernel modernos.

// resultado A

Host A — Windows Defender (AV Padrão)

⚠ Resultado: Ataque Bem-Sucedido

Sessão C2 estabelecida. O beacon conectou ao servidor de comando e controle sem nenhuma detecção ou alerta gerado pelo Defender.

Nenhum alerta. Nenhum bloqueio. O processo powershell.exe executou normalmente e a sessão do C2 foi estabelecida com sucesso.

O motivo é simples: o Defender não possuía assinatura para o payload criptografado, não há arquivo em disco para escanear, e ele não tem visibilidade sobre a transição de permissão de memória realizada via VirtualProtect.

📸 Print sugerido #1 — Sessão C2 ativa

Terminal do PowerShell no Host A após a execução — sem nenhum bloqueio ou notificação do Defender. Mostre também o console do Sliver com a sessão ativa confirmando a conexão estabelecida. Esse é o "antes" da história: ataque funcionou, sistema não viu nada.

// resultado B

Host B — Microsoft Defender for Endpoint (EDR)

🛡 Resultado: Ataque Bloqueado

Duas detecções em camadas distintas foram geradas antes de qualquer execução do shellcode. O processo foi encerrado pelo MDE antes de estabelecer conexão com o C2.

Com o MDE, o ataque não passou da fase inicial. Duas camadas de detecção agiram de forma independente:

Detecção 1 — AMSI com Análise Semântica

O MDE não trata o AMSI como um scanner de assinatura simples. Ele analisa o bloco de script completo, mesmo que strings estejam fragmentadas ou ofuscadas, e avalia a intenção lógica do código — não apenas a aparência textual dele.

  • Ameaça detectada: Trojan:PowerShell/AmsiFragBypass.E
  • Causa: O sensor reconheceu o padrão de acesso a propriedades internas do namespace System.Management.Automation usadas para desabilitar o AMSI, independentemente de qualquer ofuscação aplicada.
📸 Print sugerido #2 — Alerta no console do MDE

Console do MDE em security.microsoft.com → Incidents & Alerts → alerta com o nome Trojan:PowerShell/AmsiFragBypass.E. Destaque o severity, o processo envolvido (powershell.exe) e o timestamp. Esse print mostra que o EDR detectou a intenção do script, não uma assinatura de arquivo.

Detecção 2 — Monitoramento Comportamental de Memória

Mesmo que a primeira camada fosse contornada, o sensor de kernel do MDE monitora chamadas a APIs críticas como VirtualAlloc e VirtualProtect. Quando o powershell.exe tentou modificar permissões de memória em regiões associadas a DLLs de segurança do sistema, o sensor interveio imediatamente.

  • Ameaças detectadas: Behavior:Win32/AMSI_Patch.C e Behavior:Win32/AMSI_Patch_T.B12
  • Causa: Tentativa de tampering em regiões de memória restritas — comportamento característico de técnicas de evasão de EDR que tentam desabilitar sensores em tempo de execução.
📸 Print sugerido #3 — Process tree no alert detail

Alert detail no MDE mostrando Behavior:Win32/AMSI_Patch.C — destaque para a seção Process tree que exibe powershell.exe como processo pai e a linha de comando executada. Este é o print mais poderoso do post: ele demonstra visualmente que o EDR enxerga o comportamento interno do processo, não apenas o arquivo.

📸 Print sugerido #4 — Device Timeline

Device page no MDE → aba Timeline — mostrando a sequência de eventos: execução do PowerShell → chamadas de VirtualAlloc/VirtualProtect → detecção → processo encerrado. Esse print demonstra a profundidade de telemetria que o EDR oferece: você vê cada passo do ataque reconstruído em ordem cronológica.

// blue team

Hunting com KQL no Defender XDR

Além de bloquear o ataque, o MDE deixa rastros completos que permitem hunting proativo em toda a organização. A query abaixo correlaciona processos do PowerShell fazendo conexões externas com detecções do AMSI — exatamente o padrão observado no lab: 

// Hunting: PowerShell staging externo + detecção de bypass de AMSI
DeviceNetworkEvents
| where InitiatingProcessFileName =~ "powershell.exe"
| where RemoteIPType == "Public"
| join kind=inner (
    DeviceEvents
    | where ActionType == "AntivirusDetection"
    | where AdditionalFields has_any (
        "AmsiFragBypass",
        "AMSI_Patch",
        "Fileless"
    )
) on DeviceId, InitiatingProcessId
| project
    Timestamp,
    DeviceName,
    InitiatingProcessCommandLine,
    RemoteIP,
    RemotePort,
    ActionType,
    AdditionalFields
| sort by Timestamp desc
📸 Print sugerido #5 — Advanced Hunting com resultado

Advanced Hunting no Defender XDR com o resultado da query acima — mostre as colunas DeviceName, RemoteIP e ActionType preenchidas com os dados reais do lab. Esse print fecha o loop do post: detectamos, bloqueamos, e conseguimos caçar o padrão.

// conclusão

Conclusão

A diferença entre um AV e um EDR não é de quantidade de features — é de modelo de detecção completamente diferente.

O Defender padrão opera em um modelo reativo e baseado em assinatura: ele precisa conhecer o artefato para detectá-lo. Um payload criptografado, carregado puramente na memória, simplesmente não existe para ele.

O MDE opera em um modelo comportamental e contextual: ele não precisa conhecer o payload. Ele observa o que o processo faz — como usa memória, quais APIs chama, em qual sequência — e identifica o padrão malicioso antes que o dano seja causado.

Para times defendendo ambientes Microsoft: o onboarding do MDE não é um custo adicional, é a linha que separa "fomos comprometidos e não sabemos" de "tentaram comprometer e nós vimos em tempo real".

Um AV te protege do malware de 2015. Um EDR te dá visibilidade do Red Team de hoje.
